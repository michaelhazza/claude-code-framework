# Default PROJECT_CONTEXT framing block

This file ships the framework-default framing assumptions (spec §4, D6). Consuming repos inject
this block verbatim into `PROJECT_CONTEXT` unless they override it. See `context/README.md` for
the injection contract and override instructions.

---

## Stage

pre-production

---

## Framing assumptions

1. Pre-production is the default. No live users, no staged rollout, no feature
   flags unless the artifact explicitly calls for one. Do not flag missing
   rate-limits, monitoring dashboards, circuit-breakers, canary deploys, or
   E2E tests as blocking. They are "consider" at most.

2. Rapid evolution means light testing. Static gates over runtime tests.
   Pure-function unit tests only. No frontend / E2E / contract tests at this
   stage unless the artifact explicitly adds them. Missing tests for new pure
   functions are medium; missing E2E for new UI is not a finding.

3. Prefer existing primitives over new abstractions. The consuming project's
   architecture and guidelines docs list its canonical primitives (retry,
   rate-limit, error formatting, action proposal, policy enforcement, queue
   workers, scoped transaction helpers, scoped DB access, etc.). If the
   artifact introduces a new pattern duplicating one of these, that is a
   high/blocking finding, not a taste note.

4. Migrations ship without feature flags. A flag for a new column is dead
   weight in pre-production.

5. Mechanically tight is not the same as directionally right. The reviewer's
   job is mechanical tightness. Directional correctness is the operator's job:
   surface a suspected directional issue, do not block on it.

---

## How to override

Framing assumptions are injected, never baked into framework prompts. An app in a later
lifecycle stage (early-production, production) overrides this block at coordinator-invocation
time by providing its own `PROJECT_CONTEXT` text rather than reading from this file.

To override:

1. Copy this file's heading structure (`## Stage`, `## Framing assumptions`, and any additional
   sections from `context/README.md §Required sections`) into your own context source.
2. Replace the `Stage` value and edit the framing-assumption bullets to match your app's current
   posture.
3. Pass the resulting text to the coordinator as the `PROJECT_CONTEXT` argument.

Do NOT edit this file to reflect a per-app stage. This file is the framework default for
pre-production, pre-revenue apps. Per-app overrides live in the consuming repo, not here.

A reviewer that believes a framing assumption is wrong for the artifact in front of it must raise
`NEEDS_DISCUSSION`, not override the assumption silently.
