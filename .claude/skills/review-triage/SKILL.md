# Review-finding triage

Mined from a full corpus of ~1,900 adjudicated review logs (194 batches, four reviewer families). Calibration: ~80% of findings are accepted by mention volume (5,272 accepted vs 1,305 rejected) — reviewers are worth running, but the rejected fifth is highly patterned and largely mechanically filterable. Four clusters cover roughly two-thirds of FP volume: posture violations ~26%, stateless re-raises ~24%, hallucinated premises ~17%, altitude/scope ~9%. Residuals: convention nits, enforcement-blindness (a guard at a layer the reviewer can't see — over-flagged idempotency was the single largest rejected substantive class), and main-branch debt misattributed to the branch.

## Triage order (per finding, in this sequence)

1. Grep the factual anchor at HEAD — kills hallucinated premises (~17% of FP volume) instantly.
2. Diff against prior-round decisions and the deferred ledger — kills re-raises (~24%).
3. Check declared posture and altitude — kills posture/scope classes (~35% combined).
4. Walk all enforcement layers on the real code path — kills already-neutralized claims.
5. Only then adjudicate substance — and adjudicate the remedy separately from the diagnosis. Contradictory advice across iterations (X in round 2, not-X in round 4) marks judgment-territory: prefer documented deferral over an in-loop fix.

Near-automatic accepts (grep the anchor, then fix): cross-section contradiction/stale count in a spec; file-inventory mismatch; fabricated identifier; missing tenant GUC on a worker/webhook/boot path; missing org predicate on a PK-keyed write; SELECT-then-INSERT with no DB arbiter; nullable column in a unique key; timestamp/randomUUID idempotency key; fail-open guard shapes (`if (field) verify(field)`); built-but-never-wired when the phase promised wiring; casing/literal drift across a producer/consumer boundary; missing boundary `.max()`/`.uuid()`; missing ORDER BY tiebreaker; non-intercepting test shapes.

Near-automatic rejects (cite the rule, backlog residual value): production-hardening ceremony (flags, canaries, kill switches, SLOs, rate limits, DLQs, circuit breakers) against a declared pre-production posture; integration/E2E test demands against a pure-function + static-gate posture absent an explicit carve-out; evidence-free re-raises; new abstractions where an accepted primitive fits; signature/column-literal demands on architecture-altitude artifacts; behaviour-identical style findings; claims that grep to zero at HEAD or fail only in the reviewer's sandbox; checklists without file:line anchors; pre-existing main state unless the change makes it newly reachable.

## Verify before adjudicating — by claim type

- **"X doesn't exist / isn't exported / isn't registered / build fails"** — verify against branch HEAD in the real repo. Diff-only reviewers and sandboxed reviewers structurally produce phantom-absence claims; their own tool failures are never findings.
- **Any line-cited claim** — read the live file at the cited lines ±30 before acting. Never fix from the reviewer's quoted code: quoted snippets may be the non-authoritative path (the fast-path SELECT while the real write is a race-safe CTE), a stale upload, or a misread of unified diff format. If the claimed text is absent from the current artifact, reject as stale-context without further analysis.
- **"No transaction / no tenant scope / unguarded route"** — check the call-site framework for ambient wrappers (context-propagated transaction handles, route-group-level guards) before accepting; these are invisible in a diff hunk.
- **"Race / stuck forever / double-send"** — enumerate every fence first: recovery sweeps, unique indexes serialising writers, singleton keys, idempotency arbiters a layer down. Real only if no layer fences it. Require a concrete reachable trigger (input + call path) before promoting to blocking; downgrade speculative hardening for inputs no external caller controls (env vars, server-derived IDs).
- **DB result-shape claims** (bare array vs `{rows}` wrapper) — the repo's driver init file is the tiebreaker; DB-mocked fixtures satisfy either shape and prove nothing.
- **"Missing coverage/policy"** with no file/line citation — expect "already enforced by <index/registry/wrapper>"; verify and reject with the citation. Rounds run without diff visibility produce only this class; discount them wholesale.
- **Language/database semantics claims** ("GREATEST has a NULL bug") — verify against authoritative behaviour; a wrong fix applied to appease a false claim degrades the artifact.
- **Structural-ordering findings on documents** — require verbatim quotes of the artifact's text and the proposed alternative side by side: functionally identical = misreading, reject; concretely different = apply.

## Process-level rejections (two-thirds of all rejects)

- **Duplicates.** Stateless reviewers re-raise decided findings indefinitely — the same finding up to 5-6 rounds running, often escalated from "should fix" to "blocking" with no new evidence (still a duplicate). Keep a per-artifact decision ledger keyed by finding signature; re-raise without new evidence = auto-reject with a pointer. Injecting prior-round rejection context into the reviewer prompt measurably suppresses the class. CAUTION: a narrowed sub-case of a rejected general claim can be a real defect — compare textual scope, not just type+file; adjudicate only the new delta.
- **Deadlocked re-raise = undecided product decision.** The same finding re-raised with consistent phrasing across 4+ rounds is not a technical gap — by round 3, surface it to the operator with both paths spelled out; never autonomously resolve user-facing behaviour decisions, even under a permissive standing instruction.
- **Locked decisions.** Screen every finding against the spec, plan, ADRs, and operator locks before evaluating merit. A technically-valid suggestion that reverses a documented decision is a reject-with-citation (or a spec-amendment routed to backlog), never an in-loop fix.
- **Pre-existing code.** Diff the flagged lines against the merge base; pre-existing issues route to backlog. Exception honoured consistently: if the diff ENLARGES the blast radius of a pre-existing hole, say exactly that. And "an in-production sibling does the same thing" proves pre-existence, not correctness — verify the precedent actually works before using it to reject (a real RLS bug was once rejected on that argument and reversed a round later).
- **Wrong workflow model.** Findings contradicting documented project policy (test posture, CI-only gates, integration-branch model, pre-production framing) are rejects-with-framing-citation. Testing-posture expansion is the single most-repeated FP class: convert to fixture-based pure tests or backlog notes — but the mirror error is real too (demands to REMOVE DB-backed tests inside an explicit carve-out). Accept hardening only with a concrete correctness failure only that machinery prevents. Verify any "repo guidance says…" citation actually exists — reviewers hallucinate repo instructions. Carry explicit do-not-raise lists in reviewer prompts; external reviewers never internalise framing docs.
- **Doc-contradiction traps.** A same-session ADR that contradicts the build-committed spec weaponises doc-aware reviewers into "correcting" code against the operator's real decision — the committed spec wins until the operator says otherwise; verify every new ADR against the spec section it documents before committing, and when a reviewer's fix reverses deliberate behaviour, check ground truth first.
- **Spec text itself wrong.** Specs cite columns/enums/exports that don't exist — verify the spec's claim against the real schema before flagging a conforming implementation; adjudicate against intent, route a spec-text amendment, and check commit history for correct mid-build fixes before demanding a revert.
- **Behaviour-equivalent shape.** Judge observable behaviour under all inputs, not shape equality: spec examples, illustrative field names, and approximate thresholds are not load-bearing. Quote any "remembered" requirement verbatim from the spec; identical behaviour + security property + neighbourhood-consistent shape = pass with a note, wording mismatches route as spec amendments.
- **New-primitive proposals.** Check the accepted-primitives registry before accepting any "extract a new service/table/factory" finding — the correct form is "extend primitive X". Split combined findings: accept the clarify-existing-contract half, reject the new-primitive half.
- **Dormant by design.** Zero callers is a defect only when the current phase promised the wiring — check dormant-shell acceptance bars and blocked later chunks; record genuine gaps against the follow-up build's slug.
- **Static-gate over-flags.** Shallow grep/AST gates can't trace runtime wrappers (worker factories opening scoped transactions, aliased handles); if the invariant holds at runtime, the fix is the gate's documented suppression marker or a gate-script fix, never restructuring correct code.
- **Bookkeeping noise.** Smoke-test recordings, todo tickoffs, and persisted review-log artifacts are workflow hygiene, not correctness defects — classify separately so a functional branch isn't reported broken.
- **Out of scope.** Real-but-out-of-scope findings get a backlog entry with the finding text; findings on untracked/local files are rejected outright.
- **Altitude errors.** "Insufficient detail" against a document that deliberately defers that detail to a later phase is not a defect unless a contract-level ambiguity is shown. Mockup demo numbers are illustrative, not contracts; when an acceptance clause encodes intent via a proxy and the artifact meets the stated purpose, prefer the intent reading and flag the proxy for amendment. Reject scope-expansion redesigns and one-specific-mechanism demands when the chosen mechanism satisfies the invariant.

## Running the loop

- From round 2 onward: include the full current artifact (not a code-only diff), a "decisions made so far" block, and prior-round rationale. Pin the artifact version; in manual paste workflows verify the uploaded file is the intended one.
- Don't close after one round of all-false-positives — a fresh-context second round often finds what round 1 missed (round 1 fixates). Close when: 50%+ of a round's findings re-raise decided items, or verdict trajectory and severity improve for two consecutive rounds with only nits remaining. A round where 4+ of 5 findings are disproven on verification means stale reviewer context: terminate or restart the session rather than iterate.
- Roughly a third of iteration-N spec/plan findings are ripples from iteration N-1 edits — after applying fixes, re-grep the artifact for every restatement of each changed fact before the next round; trend matters more than count.
- In unattended loops with deferred carve-outs, converge on the new-actionable-finding curve hitting zero, not on an APPROVED verdict — stateless reviewers re-raise un-appliable carve-out findings forever with renumbered IDs; record NEEDS_REVISION with open items routed, never a silent APPROVED.
- Fixes for missing-validation findings over-correct optionality to mandatory and break previously-valid rows — mirror the canonical path's optionality when patching secondary paths. Fixes applied under review pressure to architectural findings tend to get reverted; prefer deferral.
- Different review tiers (plan, spec, implementation) catch different failure classes on the same area; never skip a later tier because an earlier one "covered it". A one-shot APPROVED downstream is evidence the upstream did its job, not a reason to have skipped the check. At least one tier per build walks the operator/workflow path, not just code-vs-spec — stuck phases, wrong-lifecycle-point audit writes, and doc/code threshold off-by-ones are invisible to code-vs-spec review.
- Run independent reviewer tracks in parallel where available — different reviewers have near-zero finding overlap on the same diff; the merged compare is the deliverable. Pair with a learning step that harvests one-side-only findings into prompt improvements.
- Auto-fix authority requires a real verification signal (lint/typecheck/tests). Prose/doc reviewers ship read-only. Security-domain findings (tenant isolation, auth, idempotency, data integrity) are NEVER auto-applied regardless of how mechanical the fix looks — key the carve-out on risk domain, not finding type. Every self-iterating fix loop carries a hard round cap with escalate-and-stop. Prompt assembly for LLM reviewers (untrusted diffs/specs in the user channel with a treat-as-data directive, never the system prompt): see the llm-integration skill.
- On "final round before freeze", the defer default inverts: deferred means never — re-judge every open item against a strict bar from its content, ignoring stale severity labels.
- After every round in an operator-driven loop, surface both "close" and "run another round" and wait; never self-decide closure. Hand the operator a copy-paste-ready next-round prompt enumerating per-finding decisions.
- Treat sub-agent claims of "fix already applied" as unverified until a grep of the actual file confirms it.
- Never inherit a prior session's unverified root-cause hypothesis — reproduce and capture the real underlying error (the wrapped `err.cause` code/message, not the generic wrapper string) before fixing; a documented guess can be wrong in a way that makes the recommended fix break working paths.
